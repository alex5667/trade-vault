# Deduplication Integration - Защита от повторной обработки сообщений

## ✅ Выполнено

### 1. Создан `services/deduper.py`
Единый модуль для дедупликации сообщений с использованием Redis `SET NX EX`.

**Компоненты:**
- `RedisDeduper`: класс для атомарной проверки и установки ключей дедупликации
- `env_int()`: helper для чтения целочисленных значений из ENV

**Принцип работы:**
```python
first = deduper.acquire(key, ttl_s=172800, value=str(int(time.time())))
if not first:
    # Дубликат - пропускаем обработку, но ACK делаем
    return True
# Первая обработка - выполняем сайд-эффекты
```

### 2. Интегрирован в `signal_performance_tracker.py`

**Добавлены методы:**
- `_signal_dedup_key()`: формирует ключ дедупликации для сигналов
  - Приоритет: `sid` (бизнес-идентификатор)
  - Fallback: `(stream, msg_id)` (технический идентификатор)
  
- `_event_dedup_key()`: формирует ключ дедупликации для событий

  - Fallback: `msg_id` (технический идентификатор)

**Dedup gates добавлены в:**
- `_process_signal_message()`: защита от повторного открытия позиций
- `_process_event_message()`: защита от повторного применения трейлинга и SL

## Как это работает

### Signals (lossless режим)

**Без dedup (проблема):**
```
1. Signal приходит в pending
2. Worker обрабатывает → открывает позицию
3. Worker падает до ACK
4. Signal остается в pending
5. Другой worker забирает (claim) → открывает позицию СНОВА (дубликат!)
```

**С dedup (решение):**
```
1. Signal приходит в pending
2. Worker проверяет dedup key (sid)
3. Ключа нет → устанавливает ключ → обрабатывает → открывает позицию → ACK
4. Worker падает до ACK
5. Signal остается в pending
6. Другой worker забирает (claim) → проверяет dedup key (sid)
7. Ключ уже есть → пропускает обработку → ACK (очищает pending)
```

### Events (lossless режим)

**Без dedup (проблема):**
```
1. TRAILING_STARTED приходит
2. Worker применяет трейлинг SL
3. Worker падает до ACK
4. Event остается в pending
5. Другой worker забирает → применяет трейлинг SL СНОВА (дубликат!)
```

**С dedup (решение):**
```
1. TRAILING_STARTED приходит
2. Worker проверяет dedup key (TRAILING_STARTED, sid)
3. Ключа нет → устанавливает ключ → применяет трейлинг → ACK
4. Worker падает до ACK
5. Event остается в pending
6. Другой worker забирает → проверяет dedup key
7. Ключ уже есть → пропускает обработку → ACK (очищает pending)
```

## Выбор ключей дедупликации

### Почему `sid` важнее `msg_id`?

**Проблема с `msg_id`:**
- `msg_id` уникален только внутри конкретного Redis Stream
- Один и тот же сигнал может прилетать повторно:
  - Повторная публикация
  - Другой stream
  - Pending replay
  
**Решение с `sid`:**
- `sid` (signal_id) — бизнес-идентификатор события
- Уникален в рамках всей системы
- Один `sid` = одно событие с точки зрения домена
- Защищает от дубликатов даже из разных streams

### Примеры ключей

**Signals:**
```python
# Если есть sid
"dedup:signals:crypto-of:BTCUSDT:1764696072000"

# Если нет sid (fallback)
"dedup:signals:signals:cryptoorderflow:BTCUSDT:1764696072000-0"
```

**Events:**
```python
# TRAILING_STARTED с sid
"dedup:events:TRAILING_STARTED:crypto-of:BTCUSDT:1764696072000"

# SL_HIT с sid
"dedup:events:SL_HIT:crypto-of:BTCUSDT:1764696072000"

# Если нет sid (fallback)
"dedup:events:msg:1764696072000-0"
```

## Тестирование

### 1. Тест на дубликаты сигналов

**Сценарий:**
1. Отправить сигнал в stream
2. Дождаться обработки (позиция открыта)
3. Отправить тот же сигнал (с тем же `sid`) снова
4. Проверить, что позиция НЕ открыта повторно

**Ожидаемый результат:**
- Первый сигнал: позиция открыта, `signals_processed++`
- Второй сигнал: позиция НЕ открыта, `dedup_signals_skipped++`

### 2. Тест на дубликаты событий

**Сценарий:**
1. Отправить `TRAILING_STARTED` в `events:trades`
2. Дождаться применения трейлинга
3. Отправить тот же `TRAILING_STARTED` (с тем же `sid`) снова
4. Проверить, что трейлинг НЕ применен повторно

**Ожидаемый результат:**
- Первое событие: трейлинг применен, `trailing_updates_applied++`
- Второе событие: трейлинг НЕ применен, `dedup_events_skipped++`

### 3. Тест на pending replay

**Сценарий:**
1. Остановить worker после обработки, но до ACK
2. Перезапустить worker
3. Worker должен забрать pending сообщение
4. Проверить, что dedup защитил от повторной обработки

**Ожидаемый результат:**
- Pending сообщение обработано (ACK)
- Сайд-эффекты НЕ выполнены повторно
- `dedup_signals_skipped++` или `dedup_events_skipped++`

## Преимущества

### 1. Защита от дубликатов
- ✅ Lossless режим безопасен: pending + reclaim не создают дубли
- ✅ Повторные публикации не создают дубли
- ✅ Защита на уровне бизнес-логики (`sid`), а не только техническом (`msg_id`)

### 2. Производительность
- ✅ Атомарная операция `SET NX EX` — O(1)
- ✅ Минимальный overhead: одна Redis операция на сообщение
- ✅ TTL автоматически очищает старые ключи

### 3. Надежность
- ✅ Работает даже при падениях worker'ов
- ✅ Работает при масштабировании (несколько consumer'ов)
- ✅ Не требует дополнительной инфраструктуры

### 4. Прозрачность
- ✅ Полная статистика пропущенных дубликатов
- ✅ Debug логи для отладки
- ✅ Легко мониторить через Redis
