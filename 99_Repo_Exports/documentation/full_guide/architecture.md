# Архитектура Scanner Infrastructure — Полный Разбор

Этот документ расширяет базовое описание архитектуры (`documentation/ARCHITECTURE.md`) и ориентирован на новые команды и архитекторов. Ниже представлены все слои системы, ключевые компоненты, соглашения по взаимодействию, профиль надёжности и точки интеграции. Читайте последовательно, чтобы получить целостное понимание.

---

## 1. Архитектурная модель

**Scanner Infrastructure** построена как событийная платформа в стиле CQRS + Event Sourcing. В качестве основной шины используется Redis (Streams, Hashes, Lists, Sorted Sets). Коммуникация между компонентами минимизирует латентность и обеспечивает идемпотентность.

### 1.1 Обозначения

- **Prod** — основное окружение с MT5 и реальными биржами.
- **Stage** — полное окружение для нагрузочных тестов; MT5 подключен в демо-режиме.
- **Dev** — локальная сборка на Docker Compose, данные можно реплейить из архивов.
- **SLA/SLO** — метрики качества обслу---

## 2. Слои архитектуры

### 2.1 Data Acquisition Layer (P4.1 t0-t1)

| Компонент                | Язык / стек    | Ответственность                                                                              | Ключевые метрики                               |
| ------------------------ | -------------- | -------------------------------------------------------------------------------------------- | ---------------------------------------------- |
| `go-worker`              | Go 1.25        | Подписки на Binance/Bybit WS. Выполняет нормализацию и проставляет `t0`.                     | `ingest_latency_t0_ms`, `tick_gap_seconds`     |
| `tick_ingest_server`     | Python/FastAPI | Принимает тики из MT5, обеспечивает DQ (Data Quality) и запись в Redis Streams.              | `dq_error_count`, `dualredis_failover_total`   |
| `market_data_janitor`    | Go             | Управляет TTL и обрезкой стримов (XTRIM) для поддержания производительности Redis.           | `janitor_trim_ops_total`                       |

**Паттерн**: **P4.1 Unified Latency Contract**. Каждый тик получает `t0` при декодировании из сокета.

### 2.2 Signal Intelligence Layer (P4.1 t2-t3)

Вместо монолитных хандлеров используется **G0-G15 Gate Pipeline**. Каждый гейт — это изолированная проверка (инвариант).

| Компонент                | Гейты       | Назначение                              | Инструменты Калибровки            |
| ------------------------ | ----------- | --------------------------------------- | --------------------------------- |
| `Tick Validation`        | G0-G1 | Проверка монотонности и DQ.             | -                                 |
| `Delta & ATR Gates`      | G2-G5 | Волатильность и всплески дельты.        | `ATRTFCalibrator`, `DnCalib`      |
| `OFConfirm Strong Gate`  | G6    | Основной фильтр подтверждения OrderFlow. | `StrongGateCalibrator`            |
| `ML Confirmation`        | G10   | Классификатор (Champion/Challenger).     | `MLGatingService`                 |
| `Edge-Cost Gate`         | G11   | Расчёт матожидания с учётом комиссий.   | `EdgeCalibrator`                  |
| `Confidence Gate`        | G12   | Интегральная оценка уверенности.        | `IsotonicRegression`              |

**Ключевые соглашения**:
- `sid` (Signal ID) сохраняется на протяжении всего t2-t5.
- Гейты могут работать в режимах: `SHADOW` (только логи), `ENFORCE` (блокировка), `MONITOR`.

### 2.3 Execution Layer (P4.1 t4-t5): Journal-First

Система перешла от простых очередей к **Journal-First** модели для обеспечения консистентности и возможности реплея.

| Компонент                   | Назначение                                                                           | Технология                                                                                    |
| --------------------------- | ------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------- |
| `orders:exec` journal       | Первичный append-only журнал всех команд исполнения.                                | Redis Stream (t4 timestamp).                                                                  |
| `BinanceExecutor`           | Высокопроизводительный Go-экзекутор. Читает журнал, исполняет REST/WS команды.      | Go 1.25, `t5` completion.                                                                     |
| `ProjectionWorker`          | Материализует состояние из журнала в БД/кеш для UI и аналитики.                      | Python AsyncIO.                                                                               |
| `BootstrapSupervisor`       | Гарантирует, что экзекутор не запустится без прогретого кеша и живого User Stream.  | Health-check gating.                                                                          |

**Паттерн**: **Lease/Fencing**. Только один экзекутор может владеть лидерством по символу через Redis `SET NX EX`.

### 2.4 Analytics & Intelligence Layer

| Компонент                    | Назначение                                                                    | Метрики Качества                                                            |
| ---------------------------- | ----------------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| `News Agent`                 | LLM (GPT-4/local) извлечение событий. Публикует `NewsTightenRecoDTO`.        | `llm_reasoning_accuracy`, `news_impact_latency`                             |
| `ML Governance`              | Мониторинг дрейфа (PSI/KS) и калибровка (ECE/Brier).                          | `feature_drift_psi`, `calibration_error_ece`                                |
| `SignalPerformanceTracker` | Аналитика трейдов V7/V8, расчет MFE/MAE/TTD.                                  | `p_edge`, `sharpe_v7`                                                       |

## 3. P4.1 Unified Latency Contract

Система отслеживает задержки на 6 критических точках:
1.  **t0**: Ingest (Go) — декодирование пакета.
2.  **t1**: Redis Write (Ingest) — публикация тика.
3.  **t2**: Signal Start (Python) — начало обработки в хандлере.
4.  **t3**: Signal Publish (Python) — публикация в журнал.
5.  **t4**: Journal Ack (Redis) — подтверждение записи в `orders:exec`.
6.  **t5**: Execution Done (Executor) — ответ от биржи / локальное исполнение.

**SLO**: Hot path (t1 -> t3) должен быть < 5мс для 99% пакетов.

## 4. Infrastructure & Redis Sharding

Инфраструктура разделена на функциональные шарды для минимизации "noisy neighbor" эффекта:
- **Shard: Market Data**: Тики и стаканы. Высокая нагрузка на запись.
- **Shard: State/Core**: Состояние позиций, журналы исполнения, риск-фильтры. Высокие требования к персистентности.
- **Shard: Analytics/Metrics**: Временные ряды и промежуточные расчеты.

**Security**: Использование **Redis ACL**. Каждому сервису выдаются минимально необходимые права (например, `reserver` для Ingest, `consumer` для Handlers).

## 6. Безопасность

1. **Аутентификация**: все HTTP маршруты, доступные вне приватной сети, требуют Bearer-токенов; трейлинговые команды дополнительно защищаются HMAC.
2. **Авторизация**: роль `mt5_executor` ограничена эндпоинтами `/orders/poll`, `/orders/ack`, `/events/mt5`.
3. **Секреты**: хранение в Vault (prod), `.env.local` (dev). Скрипт `make export-vault` синхронизирует конфиг.
4. **Сетевые политики**: Docker сети `scanner_internal` и `scanner_external`; MT5 находится в отдельной сети с firewall.
5. **Логи и PII**: персональные данные не хранятся; трейдинговые идентификаторы обрезаны до последних 4 символов в логах.

При изменении security-модели обновляйте `CONFIGURATION.md` и `operations.md`.

## 7. Наблюдаемость

- **Prometheus** собирает метрики из всех сервисов. Экспортеры:
  - Go: встроенный `/metrics`.
  - Python: `prometheus_client`.
- **Grafana**: дашборды `grafana_tick_streams.json`, `grafana_tp_trailing.json`, `grafana_signal_tracker.json`.
- **Alertmanager**: три уровня критичности (`P1`, `P2`, `P3`), маршруты в Slack/Telegram/PagerDuty.
- **Логи**: комбинированный `docker logs` + централизованный FluentBit (опционально).
- **Tracing**: на 2025-11 наблюдение ведётся без распределённого трейсинга; в `roadmap.md` запланировано внедрение OpenTelemetry.
