# Требования к данным для плана улучшений (Audit & Best Practices)

Чтобы превратить проведённый аудит исполнения сделок в детальный, проверяемый и приоритизированный план улучшений, необходимы данные следующих типов:

## 1. Статика (код/конфиги/манифесты)
Нужны для точного понимания архитектуры, контрактов сообщений, параметров риска и реальных точек входа/выхода. В пакете уже предоставлены `binance_execution.zip` и `architecture.zip`. Дополнительно требуются:
- **`crypto_orderflow_service.py` и `outbox`**: Восстановить реальную структуру, контракты DTO и зависимости.
- **CI/CD и деплой**: Dockerfiles, compose/k8s манифесты, env templates. Это позволит спланировать внедрение и откаты.
- **Политики секретов/доступов**: Документация (без самих секретов!) о RBAC, хранении ключей, audit logs.

## 2. Динамика (логи/снимки состояния/очереди/ошибки/инциденты)
Нужны для доказательной оценки работы исполнения.
- **Redis Очереди и статусы**: `orders:queue:binance`, `...:processing`, `...:dlq`, `orders:exec`, `orders:state:*`. За последние 1000–10000 сообщений.
- **Ошибки Binance и ретраи**: Логи исполнителя за 7–30 дней, особенно ошибки -4120, -1021, -2021, HTTP 429/418/503.
- **Сигнальные очереди/стримы**: `signals:crypto:raw`, `stream:signals:outbox` (от 1000 до 5000 последних записей).
- **Снимок аккаунта/рисков**: `account:snapshot:binance_usdtm` (за последние 24-48 ч).

## 3. История торговли и рыночные данные (trades/PnL, свечи/тики, latency)
Нужны для анализа применимости стратегий и оценки качества исполнения:
- **История сделок/fills/PnL**: 90–180 дней (минимум 30), формат CSV/Parquet.
- **Рыночные данные**: Свечи (1m/5m/15m) или тики (если есть) за тот же период.
- **Latency/профили**: Временные метки (enqueue_ts, dequeue_ts, api_req_ts и т.д.) за 7–30 дней.
- **Мониторинг**: Prometheus/Grafana конфигурации, alert rules.

---

## Памятка по безопасности
**СТРОГО ЗАПРЕЩЕНО ПЕРЕДАВАТЬ:** 
- Binance API keys/secrets или любые токены доступа
- Полные `.env` файлы с реальными секретами (передавать только `.env.example`)
- Любые пароли и приватные ключи

**РЕКОМЕНДОВАНО К ОБЕЗЛИЧИВАНИЮ:**
- Идентификаторы ордеров (`orderId`, `algoId`, `clientAlgoId`) лучше хэшировать.
- Финансовые объёмы и PnL можно умножить на фиксированный коэффициент для сокрытия реальных абсолютных величин.

Для выгрузки данных используйте приложенные скрипты сбора Redis (bash + redis-cli) и команды `docker logs` / `kubectl logs`. Рекомендуемый формат передачи: ZIP + шифрование (gpg/age).
