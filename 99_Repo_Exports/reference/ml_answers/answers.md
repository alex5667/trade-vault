# Ответы на вопросы по ML архитектуре

### 1. Где именно сейчас формируется feature-вектор (файл/класс), и в каком формате он уходит в модель?
**Файлы:**
- `python-worker/services/ml_confirm_gate_dynamic_v8_stack_schema_v3.py` (Класс `MLConfirmGateDynamicV8StackSchemaV3`)
- `python-worker/core/ml_feature_schema_v3.py` (Класс `MLFeatureSchemaV3`)

**Формат:**
Feature-вектор собирается методом `vectorize()` схемы признаков (например, `MLFeatureSchemaV3`). Инференс-гейт вызывает этот метод, передавая ему сырые индикаторы и контекст тика. Результатом является строго плоский одномерный вектор признаков (список Python `List[float]`). Он передается напрямую в модель через стандартный API `scikit-learn`: `model.predict_proba([x_vec])` или аналогичный (сформирована матрица 1xN).

### 2. Как сейчас строится датасет для обучения (из Redis/Timescale/NDJSON)?
**Файл:** `python-worker/ml_training_tools/build_dataset_from_inputs_tb_labels_v3_mh.py`

**Формат:** Из **NDJSON** файлов. Скрипт принимает на вход два лога в формате NDJSON (сдампленные из потоков Redis):
- `--inputs`: снимки состояния фичей OrderFlow.
- `--tb`: разметка/лейблы tick-bucket (результаты трейдов).

Данные объединяются (join) по идентификатору сессии `sid`. Скрипт выполняет извлечение признаков (flattening), добавляет one-hot энкодинг (для `direction` и `scenario`) и рассчитывает различные таргеты (utility) для нескольких горизонтов. Итоговый датасет сохраняется в эффективном формате **Parquet** для последующего обучения Pandas/Scikit-Learn.

### 3. Какие горизонты/лейблы вы используете (directional move, tp/sl, future return)?
**Файлы:**
- `python-worker/ml_training_tools/train_ml_confirm_tb_util_mh_v1.py`
- `python-worker/common/ml_labeling.py`

**Горизонты:** 
Используются несколько фиксированных горизонтов времени вперед. По умолчанию это **60000, 180000, 300000 мс** (1 мин, 3 мин, 5 мин).

**Лейблы:**
Система применяет подход, основанный на R-multiple (отношение PnL к начальному расчетному риску).
В качестве главных таргетов (utility targets) для регрессии (`Ridge`, `HistGradientBoostingRegressor`) используется `util_r_{h}` (разница между достигнутым R-multiple и ценой исполнения `exec_cost_r`). При маппинге закрытых позиций учитываются стопы/тейки (`TP/WIN/SL`).
Помимо регрессионного continuous-таргета также размечаются бинарные классификационные метки: `y_util_pos_{h}` (= 1 если `util_r > 0`) и `y_edge_{h}`.

### 4. Нужна ли поддержка “sequence models” (DeepLOB-стиль) или строго табличный GBDT/LR?
**Файл:** `python-worker/ml_training_tools/train_ml_confirm_tb_util_mh_v1.py`

В текущем виде инференс и обучение спроектированы **строго под табличные модели (GBDT/LR)**.
В скриптах обучения используются ансамблевые и линейные табличные алгоритмы (`HistGradientBoostingRegressor`, `Ridge`). Сама архитектура извлечения фичей подразумевает "stateless" обработку единичного тика (вектор параметров текущего состояния потопения ликвидности).
Внедрение последовательных моделей (Sequence models в стиле DeepLOB) в данный момент **не предусмотрено** инфраструктурой, так как модель инференса ожидает только мгновенный срез: `x_vec: List[float]` и не хранит внутреннее состояние окна, а также не принимает 2D тензоры (Time × Features).
