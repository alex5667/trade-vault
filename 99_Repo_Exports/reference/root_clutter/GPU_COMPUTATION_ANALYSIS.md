# Анализ переноса вычислений с CPU на GPU (Zero-Latency Ограничения)

## Общие выводы
Перенос вычислений на GPU эффективен только тогда, когда время выполнения самой математики на GPU перекрывает накладные расходы (overhead) на передачу данных по шине PCIe (от CPU к GPU и обратно). Для систем реального времени с жесткими требованиями к latency (задержке) правило простое: **потоковая обработка по 1 тику на GPU всегда медленнее, чем на CPU.**

Исследование кодовой базы (`python-worker`) показало, что проект уже имеет `GPUService` (`cupy` / `torch` fallback) для батчевых вычислений метрик потока ордеров (`compute_obi_metrics_batch` и `compute_robust_zscore_mad`). 

Чтобы ускорить систему без потери скорости и повышения latency, необходимо применять GPU только там, где можно обеспечить векторизацию (batching) или где данные могут **постоянно жить в памяти GPU**.

## Что МОЖНО и НУЖНО перенести на GPU (Без потери скорости)

### 1. Векторные операции над большими окнами (Rolling Windows)
Если мы храним исторические буферы (например, последние 10,000 свечей или тиков) напрямую в памяти GPU (используя кольцевые буферы `cupy.ndarray` или `torch.Tensor`), то расчеты сложных статистик становятся моментальными.
- **Где в коде**: `core/tb_labeling.py` (функции типа `barrier_stats`, которые сейчас частично ускорены через `@njit`), `microstructure` расчеты.
- **Как**: Обновлять 1 элемент в кольцевом буфере GPU дешево. Пересчитывать агрегацию (Z-score, IQR, квантили) по всему вектору на GPU гораздо быстрее, чем на CPU, и мы просто забираем скалярный результат (офлоад обратно на CPU).

### 2. Поддержка и анализ L2 Order Book (DOM)
Анализ стакана (`SmartClusterAnalyzer.analyze_from_dom`, `l2_microstructure_engine.py`) включает суммирование, поиск дисбалансов (imbalances) и расчет профилей объема (Volume Profile).
- **Почему GPU**: Если количество уровней стакана велико (например, 1000+ уровней), матричные операции над слепками стакана (свертки для поиска плотностей, агрегация объемов) идеально параллелятся.

### 3. Периодическая калибровка и ML Обучение
Процессы, которые работают в фоне (или при старте) и не лежат в критическом пути каждого тика.
- **Где в коде**: `train_confidence_calibrator_v2.py` (использует `np.linalg.solve`), `feature_selection_loop_v1.py`.
- **Эффект**: Использование `cupy.linalg.solve` или `torch` для этих задач высвободит процессорное время основного потока (CPU) под обработку тиков, убрав спайки latency в основном процессе.

## Что НЕЛЬЗЯ переносить на GPU (Усугубит Latency)

### 1. LightGBM Инференс одиночных сигналов (Batch Size = 1)
- **Где в коде**: `confidence_scorer.py` -> `model.predict([f_vec])[0]`
- **Почему**: Обход логического дерева на процессоре занимает микросекунды. Копирование одного вектора `f_vec` из RAM в VRAM, запуск ядра GPU и копирование вероятности `ml_prob` обратно через PCIe будет занимать в 10-50 раз больше времени (миллисекунды). GPU-инференс LightGBM/XGBoost целесообразен только при размере батча от 10,000+ строк.

### 2. Скалярные обновления индикаторов (EMA, SMA, ATR)
- Обновление состояния индикаторов тик-за-тиком формулой `EMA_new = alpha * price + (1-alpha) * EMA_old` является сугубо последовательным процессом (зависит от предыдущего шага). На GPU это вызовет катастрофические задержки без какого-либо ускорения.

### 3. Парсинг данных и сериализация
- Чтение из Redis, распаковка JSON/MsgPack, подготовка словарей (Dict) - это задачи сугубо для CPU.

## Рекомендации для архитектуры
Для максимальной утилизации GPU с нулевым эффектом на latency, архитектура должна быть следующей:
1. **CPU Worker (Go / Python)** читает стрим и поддерживает актуальное состояние в памяти RAM.
2. При поступлении пачки изменений, CPU асинхронно отправляет вектор обновлений в **GPU Worker** (или общую разделяемую память).
3. При запросе сигнала (Signal trigger), CPU использует легкие предварительно рассчитанные метрики. Тяжелая математика крутится на GPU обособленно и возвращает агрегаты (например, Regime State) обратно на CPU с частотой 10-100 мс, а не на каждый тик.
