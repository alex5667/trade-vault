# Выводы по аудиту реализации 4-этапного плана Confidence

**Общий вывод**: План в значительной степени реализован. Основной функционал калибровки, непрерывных бонусных метрик, ML-слоя и исправления дрифта ключей (sweep) присутствует в коде. Существуют незначительные расхождения в нейминге по Изменению A.

Ниже приведена детальная разбивка каждого пункта плана на основе текущего состояния файлов в `python-worker/` (особенно `confidence_scorer.py`, `signal_confidence.py`, `tick_processor.py`, `orderflow_strategy.py`).

### Изменение A: контракт confirmations (v1 → v2)
- **Статус**: Частично реализовано (используется другой нейминг).
- **Детали**: В коде отсутствует явное использование названия `confirmations_map`, однако вместо него повсеместно внедрено использование словаря `ctx.evidence` (в котором хранятся значения float). Обратная совместимость поддерживается: парсятся старые строки вида `k=v` из списка `ctx.confirmations`. Значения корректно приводятся к числовым типам, как и требовал план.

### Изменение B: устранить «мертвые» confirmations из-за несовпадения ключей
- **Статус**: Реализовано.
- **Детали**: В `tick_processor.py` логика корректно вычисляет `eq_kind` и явно добавляет канонические ключи `sweep_eqh=1` или `sweep_eql=1` в `confirmations` и `indicators`. Оставлен фоллбэк `sweep=1`.
В скорере (`confidence_scorer.py`/`signal_confidence.py`) внедрена логика проверки `sweep_eqh`, `sweep_eql`, а также `sweep` с алиасами (например, `if _ctx_confirm_value(ctx, "sweep") > 0 ... s_sweep = 0.5`). 
Аналогично поддерживаются алиасы для айсбергов: `ice_strict` и `iceberg_strict` взаимно заполняют друга (`flags.setdefault...`).

### Изменение C: сделать contributions непрерывными там, где уже есть strength
- **Статус**: Реализовано.
- **Детали**: В оценке дивергенции учитывается непрерывный мультипликатор `div_strength_mult = min(max(div_strength / ref, 0.8), 1.3)`, основанный на `div_strength`. Для RSI величина сигнала (`rsi_val`) используется внутри `_clamp(rsi_val, 0, 1)`, сохраняя непрерывность вместо бинарного 0/1.

### Изменение D: добавить калибровочный слой поверх rule-скоринга
- **Статус**: Реализовано.
- **Детали**: В `orderflow_strategy.py` используется инстанс `ConfidenceCalibratorBundleRuntime`, применяемый для калибровки сырых скоров. Сырой скор перед калибровкой фиксируется (например, `indicators["confidence_raw"] = indicators.get("confidence_v1")`). В результате формируется калиброванная вероятность (`confidence_pct`/`confidence_ratio`), что полностью соответствует замыслу оффлайн-калибровки поверх rule-скоринга.

### Изменение E: ML-слой (опционально)
- **Статус**: Реализовано.
- **Детали**: Конвеер поддерживает ML-вероятности (late fusion). В `tick_processor.py` извлекается `ml_p_cal` для Decision Records, а в `confidence_scorer.py` (блок "Phase3: optional ML blend") происходит взвешенное смешивание (`logit blend`) сырого скора и `ml_p`, если `ml_p` (он же `ml_p_cal`) доступен в `ctx.evidence`. Влияние ML-слоя (alpha) динамически регулируется.
