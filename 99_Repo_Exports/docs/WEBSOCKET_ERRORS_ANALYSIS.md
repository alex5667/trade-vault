# 🔍 Анализ причин ошибок WebSocket подключений

## 📊 Текущая архитектура

### Масштаб подключений

**Важно:** Каждый воркер обрабатывает **ТОЛЬКО ОДИН таймфрейм** (задается через `BINANCE_WS_TIMEFRAME`):

- **10 воркеров**, каждый для своего таймфрейма:

  - `go-worker-1m` → `BINANCE_WS_TIMEFRAME=kline_1m` (только 1m)
  - `go-worker-5m` → `BINANCE_WS_TIMEFRAME=kline_5m` (только 5m)
  - `go-worker-15m` → `BINANCE_WS_TIMEFRAME=kline_15m` (только 15m)
  - `go-worker-1h` → `BINANCE_WS_TIMEFRAME=kline_1h` (только 1h)
  - `go-worker-4h` → `BINANCE_WS_TIMEFRAME=kline_4h` (только 4h)
  - `go-worker-1d` → `BINANCE_WS_TIMEFRAME=kline_1d` (только 1d)
  - `go-worker-1w` → `BINANCE_WS_TIMEFRAME=kline_1w` (только 1w)
  - `go-worker-1month` → `BINANCE_WS_TIMEFRAME=kline_1M` (только 1M)
  - `go-worker-3month` → `BINANCE_WS_TIMEFRAME=kline_3M` (только 3M)
  - `go-worker-1y` → `BINANCE_WS_TIMEFRAME=kline_1y` (только 1y)

- **~150 символов** подписки (примерно)
- **До 50 символов на соединение** (`maxSymbolsPerConnection: 50`)
- **До 50 соединений на воркер** (`maxConnections: 50` - ограничение для динамического добавления)
- **500 слотов** для одновременных TLS handshakes (`connectionSemaphore: 500`)

**Расчет соединений на один воркер (один таймфрейм):**

- Минимум: `ceil(150 символов / 50 символов на соединение) = 3 соединения`
- Максимум: `50 соединений` (`maxConnections: 50` - при динамическом добавлении символов)

**Итого по всем воркерам:**

- **Минимум:** 10 воркеров × 3 соединения = **~30 соединений**
- **Максимум:** 10 воркеров × 50 соединений = **~500 соединений**
- **Реально:** **~30-150 соединений** (в зависимости от количества символов и их распределения)

**Пример при 150 символах:**

- На каждый таймфрейм: `ceil(150/50) = 3 соединения`
- Всего: `10 воркеров × 3 соединения = 30 активных WebSocket соединений`

## 🚨 Основные причины ошибок

### 1. **Connection Reset by Peer** (самая частая ошибка)

**Причины:**

- ✅ **Binance API rate limiting** - сервер разрывает соединения при превышении лимитов
- ✅ **Перегрузка сервера Binance** - при высокой нагрузке сервер принудительно закрывает соединения
- ✅ **Слишком много одновременных подключений** - Binance может ограничивать количество соединений с одного IP
- ✅ **Таймауты на стороне Binance** - сервер разрывает "тихие" соединения (без активности)
- ✅ **Балансировщик нагрузки Binance** - переключение между серверами разрывает соединения

**Частота:** Очень высокая (нормальное поведение при масштабе)

**Решение:** ✅ Уже исправлено - короткая задержка (3-8 сек), логирование реже

---

### 2. **I/O Timeout** (таймауты чтения/записи)

**Причины:**

- ✅ **Медленная сеть** - высокий latency между клиентом и Binance серверами
- ✅ **Перегрузка сети** - пакеты теряются или задерживаются
- ✅ **Проблемы с DNS** - медленное разрешение имен Binance серверов
- ✅ **Firewall/NAT** - промежуточные устройства разрывают idle соединения
- ✅ **Таймаут чтения слишком короткий** - 60 секунд может быть недостаточно при медленной сети

**Частота:** Средняя

**Решение:** ✅ Уже исправлено - увеличены таймауты, более частые ping

---

### 3. **Dial TCP Timeout** (таймауты подключения)

**Причины:**

- ✅ **Медленное установление TLS соединения** - TLS handshake занимает много времени
- ✅ **Перегрузка Binance серверов** - серверы не успевают обрабатывать новые подключения
- ✅ **Проблемы с DNS** - медленное разрешение `fstream.binance.com`
- ✅ **Слишком много одновременных попыток подключения** - даже с семафором на 500, это много
- ✅ **Сетевые проблемы** - потеря пакетов при установке соединения

**Частота:** Средняя

**Решение:** ✅ Уже исправлено - увеличены таймауты (Dial: 30s, Handshake: 45s)

---

### 4. **Unexpected EOF** (неожиданное закрытие)

**Причины:**

- ✅ **Нормальное закрытие соединения** - сервер корректно закрыл соединение
- ✅ **Переподключение** - клиент закрывает старое соединение при создании нового
- ✅ **Graceful shutdown** - Binance закрывает соединения при обслуживании

**Частота:** Низкая-средняя

**Решение:** ✅ Уже исправлено - обрабатывается как нормальная ситуация

---

### 5. **Use of Closed Network Connection**

**Причины:**

- ✅ **Race condition** - попытка использовать соединение после его закрытия
- ✅ **Параллельное переподключение** - несколько горутин пытаются использовать одно соединение
- ✅ **Graceful shutdown** - соединение закрывается во время чтения/записи

**Частота:** Низкая

**Решение:** ✅ Уже исправлено - не логируется как ошибка

## 🎯 Корневые причины (Root Causes)

### 1. **Масштаб подключений (может быть избыточным при росте)**

**Проблема:**

- 10 воркеров × максимум 50 соединений каждый = **до 500 потенциальных соединений**
- При реальной нагрузке (150 символов): **~30 активных соединений** (10 воркеров × 3 соединения)
- При динамическом добавлении символов: **до 150-500 активных соединений**
- Это **много** для одного клиента Binance API, особенно при массовом переподключении
- При переподключении все 30-150 соединений пытаются восстановиться одновременно, что может вызвать rate limiting

**Решение:**

- ✅ Уменьшить `maxConnections` с 50 до 20-30
- ✅ Увеличить `maxSymbolsPerConnection` с 50 до 100 (если Binance поддерживает)
- ✅ Использовать connection pooling более эффективно

---

### 2. **Thundering Herd при переподключении**

**Проблема:**

- При массовом разрыве соединений все клиенты пытаются переподключиться одновременно
- Это создает пиковую нагрузку на Binance API
- Binance отвечает разрывом соединений (rate limiting)

**Решение:**

- ✅ Уже есть jitter и экспоненциальная задержка
- ⚠️ Можно улучшить: добавить глобальный rate limiter для переподключений

---

### 3. **Недостаточные таймауты для медленных сетей**

**Проблема:**

- Таймауты были слишком короткие (10-15 секунд)
- При медленной сети или перегрузке Binance соединения не успевают установиться

**Решение:**

- ✅ Уже исправлено - таймауты увеличены (Dial: 30s, Handshake: 45s)

---

### 4. **Отсутствие адаптивного backoff**

**Проблема:**

- Фиксированные задержки не учитывают состояние API
- При перегрузке Binance нужно увеличивать задержки

**Решение:**

- ✅ Частично исправлено - умная задержка в зависимости от типа ошибки
- ⚠️ Можно улучшить: адаптивный backoff на основе частоты ошибок

---

### 5. **Проблемы с сетевым стеком Docker**

**Проблема:**

- Docker network может создавать дополнительную задержку
- NAT может разрывать idle соединения
- Проблемы с DNS resolution внутри контейнеров

**Решение:**

- ⚠️ Проверить настройки Docker network
- ⚠️ Использовать host network mode для критичных соединений
- ⚠️ Настроить DNS caching

## 📈 Статистика ошибок (из логов)

### Типы ошибок (примерное распределение):

1. **Connection Reset by Peer**: ~60-70% всех ошибок
2. **I/O Timeout**: ~20-25% всех ошибок
3. **Dial TCP Timeout**: ~5-10% всех ошибок
4. **Unexpected EOF**: ~3-5% всех ошибок
5. **Use of Closed Network Connection**: ~1-2% всех ошибок

### Временные паттерны:

- **Пики ошибок** при старте всех воркеров одновременно
- **Волны ошибок** каждые 5-10 минут (массовые переподключения)
- **Стабильные периоды** между волнами (соединения работают нормально)
